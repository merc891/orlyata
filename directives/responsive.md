# Директива: Адаптивный дизайн

Правила, которые должны соблюдаться при любой работе с CSS на сайте. Перед правкой или ревью адаптива — пройди по чек-листу в конце.

---

## Принципы

### 1. Единый текстовый стиль на всех брейкпоинтах

**Правило:** Один элемент = один текстовый стиль во всех вьюпортах. Разрешено менять только `font-size`. `line-height`, `letter-spacing`, `font-family`, `font-weight` — не меняются между брейкпоинтами.

**Технически обеспечено:**
- `--text-*-line` токены — **unitless ratio** (`--text-display-line: 1`, `--text-xl-line: 0.8`, …), а не пиксели. При смене `font-size` line-height масштабируется автоматически, ratio сохраняется.
- В `@media` блоках никогда не пишем `line-height`/`letter-spacing` override — только `font-size`.

**Запрещено:**
```css
/* ❌ смена стиля — разные line-height и letter-spacing */
@media (max-width: 1279px) {
  .heading {
    font-size: var(--text-headline-size);
    line-height: var(--text-headline-line);   /* другой стиль! */
    letter-spacing: var(--text-headline-track); /* другой стиль! */
  }
}
```

**Правильно:**
```css
/* ✅ только размер меняется */
@media (max-width: 1279px) {
  .heading {
    font-size: var(--text-headline-size); /* 64px */
    /* line-height и letter-spacing наследуются из базы */
  }
}
```

**Почему:** Если на 1280+ используется `Heading/Hero` (ratio 0.65), то на 1279 и ниже тоже должен быть `Heading/Hero` — только меньшего размера. Нельзя на 1280+ ставить `h1`-стиль, а на 1279 — `h2`-стиль.

---

### 2. Сохранение визуальной иерархии

**Правило:** Иерархия шрифтов с десктопа сохраняется на каждом брейкпоинте — как внутри компонента, так и между компонентами страницы (кросс-компонентная иерархия). Если A крупнее B на 1280px — A обязан оставаться крупнее B на 1279px и на 767px.

**Нельзя допускать:** два элемента разного уровня иерархии становятся одинакового размера после адаптации.

#### 2.1. Внутри компонента
**Пример нарушения — Hero, мобиль (до фикса):**
```
heading     → --text-xl-size (36px)
eventsLabel → --text-xl-size (36px)
```
Оба по 36px → иерархия сломана.

#### 2.2. Между компонентами
**Пример нарушения — главная, мобиль (до фикса):**
```
Hero brand "хоровая капелла мальчиков орлята"  → 36px
PhotosVideos heading "медиагалерея"            → 36px
Notes heading "ноты"                           → 36px
```
Главный заголовок (бренд) сжался до уровня обычных section headings. Решение: бренд должен оставаться минимум на одну ступень крупнее.

#### 2.3. Решение коллизий через layout, а не типографику
Если две сущности **должны** иметь одинаковый размер по смыслу (например, числовая статистика и её заголовок) — коллизия снимается **разделением через layout**: разные колонки, плашки, контрастные фоны. Не нужно искусственно понижать одну в типографике, если это ломает её роль.

**Пример — Stats на мобиле:** `.title` («в цифрах и фактах») и `.value` (числа) оба `--text-xl-size`. Решено через сетку 2×2 для cells + titleCell на всю ширину сверху — числа отделены плашками, а не размером шрифта.

---

### 3. Одна ступень token-шкалы за брейкпоинт

**Правило:** Между соседними брейкпоинтами `font-size` элемента смещается не более чем на одну ступень token-шкалы. Не перепрыгиваем через ступени.

**Шкала (от большего к меньшему):**
```
96px  --text-hero-size
64px  --text-headline-size
44px  --text-display-size
36px  --text-xl-size
24px  --text-lg-size
18px  --text-body-lg-size
16px  --text-body-md-size
14px  --text-body-sm-size
```

**Примеры корректной адаптации:**
| Элемент | Desktop | Tablet ≤1279 | Mobile <768 |
|---|---|---|---|
| Hero brand | 96 (hero) | 64 (headline) | 64 (headline) |
| EnrollCTA heading | 64 (headline) | 64 (headline) | 44 (display) |
| Footer tagline | 64 (headline) | 64 (headline) | 44 (display) |
| Section heading | 44 (display) | 44 (display) или 36 | 36 (xl) |
| Hero eventsLabel | 44 (display) | 36 (xl) | 24 (lg) |
| History.lead | 44 (display) | 44 (display) | 36 (xl) |

**Почему:** Перепрыгивание (96 → 36) ломает плавность пирамиды и разрывает связь между десктопным и мобильным образом. Один шаг — это естественное «сжатие» того же стиля.

---

### 4. Компенсация утерянных layout-распираний

**Правило:** Когда элемент при переходе на узкий брейкпоинт теряет `flex: 1 1 auto` или `height: 100%` распирание (родитель стал `height: auto`) — вместо распирания задаём явный `margin`/`padding`, чтобы сохранить визуальный ритм.

**Пример — Hero:**
- Desktop: `.heading` имеет `flex: 1 1 auto` и тянется на всю высоту tile до блока событий
- ≤1279px: `.tile { height: auto }` — распирание исчезло → добавлен `margin-bottom: 120px` на `.heading`
- <768px: `margin-bottom: 80px`

**Симптом:** заголовок и следующий блок «прилипли» друг к другу после перехода на узкий вьюпорт. Это сигнал что компенсация нужна.

|---|---|---|

## Пирамида уровней на главной (mobile <768px)

Эталонная пирамида для главной страницы. При добавлении новых компонентов их типографика должна вписываться в эту шкалу.

```
64px  Hero brand                                      --text-headline-size
        - «хоровая капелла мальчиков орлята»

44px  Leads / page statements                         --text-display-size
        - EnrollCTA «Хотите вырастить творческую…»
        - Footer tagline «Сегодня орлята, а завтра — орлы»

36px  Section headings + section-level paragraphs    --text-xl-size
        - «медиагалерея», «ноты», «в цифрах и фактах», …
        - History.lead «Создана руководителями…»
        - Stats title + Stats value (разделены layout-ом, см. 2.3)

24px  Block-level titles                              --text-lg-size
        - Hero eventsLabel «события»
        - PhotosVideos featured/sideTitle

18px  Body-lg                                         --text-body-lg-size
16px  Body-md                                         --text-body-md-size
14px  Body-sm                                         --text-body-sm-size
```

**Принцип согласованности:** компоненты соседних уровней (например, две статьи или два section headings) — должны использовать **один и тот же токен**. Не допускается, чтобы один section heading был 44px (display), а другой 36px (xl) на одном и том же брейкпоинте.

## Чек-лист аудита адаптива

Перед коммитом или ревью адаптива пройди по списку для целевого брейкпоинта (обычно начинаем с самого узкого).

### 1. Кросс-компонентный аудит размеров
- Открой все CSS-модули компонентов страницы
- Выпиши `font-size` для каждого text-элемента на целевом брейкпоинте (учитывай каскад: base + media queries)
- Сведи в таблицу: компонент → элемент → font-size
- Проверь что значения образуют строгий нисходящий ряд по пирамиде уровней

### 2. Проверка иерархии
- Page brand > Leads > Section headings > Block titles > Body
- Внутри каждого уровня — все элементы используют один и тот же токен
- Нет коллизий «A = B» где A и B разных уровней

### 3. Проверка правила #1
- В `@media` блоках есть **только** `font-size` overrides (для text-стилей)
- `line-height`, `letter-spacing`, `font-family`, `font-weight` НЕ переопределяются
- Исключение: первое объявление стиля для элемента, который не существует вне media-query (например `.nav a` в мобильном меню Sidebar)

Скрипт для быстрой проверки:
```bash
cd app/src && for f in $(find components -name "*.module.css"); do
  awk '/@media/{m=1} m&&/}/&&!i{m=0} m&&/{/&&!/@media/{i++} m&&/}/&&i{i--} m&&(/line-height/||/letter-spacing/){print FILENAME":"NR": "$0}' "$f"
done
```

### 4. Проверка правила #3 (одна ступень)
- Для каждого элемента сравни font-size на соседних брейкпоинтах
- Допустимо: токен не меняется, или меняется на соседний по шкале
- Запрещено: перепрыгивание через ступени (например 96px → 36px напрямую)

### 5. Проверка компенсаций (правило #4)
- Найди элементы с `flex: 1 1 auto` или `height: 100%`
- На каждом брейкпоинте проверь: распирание сохраняется или родитель стал `height: auto`?
- Если распирание потеряно — должен быть `margin`/`padding` компенсация

### 6. Визуальная проверка
- Запусти dev-сервер и используй DevTools для перехода между брейкпоинтами
- Или вызови skill `responsiveness-check` для автоматического снимка

## Что категорически запрещено

- Хардкод `font-size`, `line-height`, `letter-spacing` в `px` (за пределами `tokens.css`)
- `clamp()` с произвольными значениями (только если нужный токен отсутствует — добавь его в `tokens.css`)
- Менять `line-height`/`letter-spacing` в `@media` блоках (см. правило #1)
- Использовать разные text-стили для одного семантического уровня иерархии в разных компонентах
- Перепрыгивать через ступени token-шкалы за один брейкпоинт
